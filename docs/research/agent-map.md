# goodvibes-agent (@pellux/goodvibes-agent v1.6.0) — Codebase Map

Bun TUI app (`bun >= 1.3.10`), single binary `goodvibes-agent` (`bin/goodvibes-agent.ts` → `src/main.ts`). ~180 model-tool modules in `src/tools/`, workspace UI in `src/input/agent-workspace-*.ts`, product brain in `src/agent/`. SDK pinned: `@pellux/goodvibes-sdk@1.3.1` (devDependency; bundled at build — `package.json`). Positioning (`README.md`): "installable autonomous operator assistant" — one assistant that can chat, plan, remember, research, schedule, send, generate, run visible agents, and operate the GoodVibes daemon contract with confirmation gates.

## 2. Differences From / Overlap With goodvibes-tui

Both are Bun terminal apps on the same SDK (tui@1.10.0 pins sdk 1.3.2; agent@1.6.0 pins sdk 1.3.1) and share the renderer, input, command-registry, config, permission, provider, and session foundations.

| Dimension | goodvibes-tui (`/home/buzzkill/Projects/goodvibes-tui/README.md`) | goodvibes-agent |
|---|---|---|
| Product | Vibecoding harness: coding, WRFC review loops, git/worktrees, panels/control rooms, eval/replay/forensics, LSP/tree-sitter intelligence, marketplace/plugins, sandbox/QEMU | Personal operator assistant: chat-first, Personal Ops (email/calendar/tasks), research, documents, routines/personas/skills, knowledge, autonomy with receipts |
| Daemon | **Hosts** the daemon/HTTP listener (`daemon.enabled` default true, loopback :3421; control-plane REST/SSE/WS/web shell — README "Local daemon and HTTP listener") | **Client only.** Connects to externally managed host; blocks all `daemon/serve/listener` CLI commands; cannot start/stop/expose the host (`src/cli/parser.ts:61-80`, `README.md:183-202`) |
| Automation execution | Runs local scheduled jobs, hooks, cron-like agent tasks | Local automation execution **disabled by design** (`src/runtime/bootstrap.ts:649-651`); schedules live in the daemon |
| Agents | In-process agents, archetypes, WRFC controller, worktree isolation | Visible serial-by-default subagents + work-plan dispatch; delegates heavy build work to TUI via shared sessions |
| Knowledge | Default knowledge backend (GraphQL, projections, connectors) | **Isolated** `/api/goodvibes-agent/knowledge/*` only; fails closed on scope contamination (`src/agent/knowledge-scope-alias.ts`) |
| Unique to TUI | Panels/Fleet control room, six consoles, REPL/eval runtimes, LSP intelligence, evaluation/replay/incident forensics, marketplace, QEMU sandbox, distributed peer/node-host runtime, omnichannel hosting | — |
| Unique to Agent | VIBE.md, personas, skills, routines, profiles+starter templates, learning curator + consolidation receipts, Personal Ops (native IMAP/SMTP + ICS calendar), unified inbox, away digest, research runs/sources/reports, Documents & Compare + blind model compare, review packet wizard, setup wizard w/ checkpoints, route planner (`route`), settings import *from* TUI | — |
| Overlap (shared SDK services both instantiate) | providers/model routing/failover, MCP, permissions/policy, secrets, sessions, artifacts, channels (TUI hosts; Agent consumes), voice/TTS, web search, memory, hooks, bookmarks, keybindings | same (`src/runtime/services.ts:404-...` mirrors TUI's service graph) |

Settings import bridge: Agent imports provider selections, UI prefs, permissions, subscriptions, surfaces, tools, daemon endpoint from goodvibes-tui stores (preview→confirm, redacted, source store never mutated) (`settings action:"import"`, `src/tools/` import tool; `README.md:8`).

## 3. Architecture

**Process model: one process.** `src/main.ts` renders raw-ANSI alt-screen TUI (Compositor/UIFactory in `src/renderer/`), runs the SDK `Orchestrator` in-process (`src/core/orchestrator.ts` is a pure re-export of `@pellux/goodvibes-sdk/platform/core`), and builds the entire service graph in `createRuntimeServices()` (`src/runtime/services.ts` — ~100 SDK services: ProviderRegistry, ToolRegistry, PermissionManager, AgentManager/AgentOrchestrator/WrfcController, AutomationManager (observability-only), MemoryStore/MemoryRegistry/MemorySpineClient, KnowledgeService (agent-scoped), ArtifactStore, ChannelPluginRegistry/ChannelDeliveryRouter, WatcherRegistry, ApprovalBroker, SharedSessionBroker, SessionSpineClient, McpRegistry, SecretsManager, ProcessManager, FileUndoManager, HookDispatcher, PluginManager, VoiceService, WebSearchService, MultimodalService, RemoteSupervisor, ReplayEngine, etc.). Bootstrap composition root: `src/runtime/bootstrap.ts` + `bootstrap-core.ts`.

**SDK usage (concrete):**
- Deep-subpath imports throughout; top import counts (non-test): `platform/config` 230, `platform/tools` 211, `platform/providers` 201, `platform/state` 122, `platform/types` 108, `platform/utils` 88, `platform/hooks` 57, `platform/artifacts` 56, `platform/automation` 53, `platform/core` 52, plus `platform/permissions|knowledge|control-plane|channels|mcp|agents|sessions|security|runtime/memory-spine|intelligence|media|voice|calendar|pairing|watchers|scheduler|...` and `contracts`, `events`, `browser`.
- `ConfigManager`, `DEFAULT_CONFIG`, `CONFIG_SCHEMA` from `platform/config` (`src/config/index.ts`).
- Typed daemon client: `createBrowserGoodVibesSdk` from `@pellux/goodvibes-sdk/browser` + `OperatorMethodInput/Output<'automation.schedules.create'>` etc. from `@pellux/goodvibes-sdk/contracts` — used in `src/agent/routine-schedule-promotion.ts`, `reminder-schedule.ts`, `autonomy-schedule.ts`, `schedule-edit.ts` (`automation.jobs.update`), `operator-actions.ts` (`approvals.approve|deny|cancel`, `/api/automation/jobs/{id}/run|enable|disable`, `/api/automation/runs/{id}/cancel|retry`).
- Session spine + memory spine REST transports: `src/runtime/session-spine-rest-transport.ts`, `memory-spine-rest-transport.ts`; `MemorySpineClient` starts LOCAL (embedded, offline-required) and activates CLIENT mode over REST when a daemon is adopted — single-writer invariant, never touches local store afterward (`src/runtime/services.ts:659-690`).

**Daemon relationship: same daemon as goodvibes-tui.** Default `http://127.0.0.1:3421` (`src/cli/endpoints.ts:7`, config `controlPlane.host/port`), overridable via `--runtime-url` / `GOODVIBES_AGENT_RUNTIME_URL` (legacy `GOODVIBES_AGENT_BASE_URL`). That's the control-plane host that the goodvibes-tui package installs/runs. Auth token at `~/.goodvibes/daemon/operator-tokens.json`; `setup action:"token"` repairs it via SDK pairing helper. Agent runs its own **conversation brain locally** (Orchestrator + providers in-process) but treats the daemon as its **capability runtime** (schedules, watchers, approvals, automation, channels delivery, knowledge routes, remote execution). Fully offline-tolerant: local memory/sessions/registries work with no daemon.

**Storage.** Agent home = `GOODVIBES_AGENT_HOME` or `~` with surface root `agent` (`GOODVIBES_AGENT_SURFACE_ROOT = 'agent'`, `src/config/surface.ts`), i.e. `~/.goodvibes/agent/` via `ShellPathService.resolveUserPath`. Files observed: `keybindings.json`, `subscriptions.json`, `services.json` (project), `model-limits.json`, favorites/benchmark stores, `providers/*.json`, `control-plane/approvals.json`, `control-plane/sessions.json` (project-scoped), per-registry JSON stores (routines/personas/skills/notes), profile homes for named profiles. Memory: **one canonical cross-surface SQLite** (`resolveCanonicalMemoryDbPath(home)` from `platform/state`) shared with the TUI so facts recall across products; legacy `agent/memory.sqlite` folded in at boot without deletion (`src/runtime/services.ts:659-663`). Code index at `<workspace>/.goodvibes/agent/code-index.sqlite`. Vector search via `sql.js` + `sqlite-vec`.

## 4. Proactive Machinery

Design principle: **"visible autonomy"** — no hidden background jobs; every long-running thing has a user-readable task, status, cancellation route, success criteria, confirmation gates.

- **Execution lives in the daemon.** Local automation startup is hard-disabled (`src/runtime/bootstrap.ts:649-651` logs "Local automation startup is disabled in GoodVibes Agent; use connected-host observability instead"). Schedules/reminders/autonomous jobs are created on the connected host via typed operator methods: `automation.schedules.create` (POST `/api/automation/schedules`) for routine promotion (`src/agent/routine-schedule-promotion.ts`), reminders (`src/agent/reminder-schedule.ts`), autonomous schedules (`src/agent/autonomy-schedule.ts`); edits via `automation.jobs.update` (`schedule-edit.ts`); lifecycle run/enable/disable/cancel/retry via `operator-actions.ts`. Schedule kinds: `cron | every | at`, with timezone, delivery targets (webhook/surface/integration/link across 16 surface kinds incl. slack/discord/telegram/signal/whatsapp/telephony). Confirmed actions record redacted local receipts (`routine-schedule-receipts.ts`).
- **Triggers/watchers** — daemon-owned `watchers.create/update/run/start/stop/delete` operator methods for webhooks, Gmail/email connector triggers, event streams; Agent's `autonomy action:"intake"` classifies an ongoing-work ask into schedule vs watcher vs process route and enforces trusted source/scope + success criteria + confirmation (`docs/tools-and-commands.md:200`).
- **Autonomy queue** — `autonomy action:"queue|item"` normalizes work-plan items, research runs, connected tasks, approvals, automation runs, schedules, reminders, routine promotions, delegated agents, deliveries into cards with live status, log tails, and exact pause/resume/cancel/approve/deny/retry routes (`src/tools/agent-harness-autonomy-*.ts`).
- **Routines** are local recipes, not schedulers: starting one prints steps into the conversation and bumps `startCount`; only explicit confirmed *promotion* creates a daemon schedule.
- **Local ambient proactivity** (in-process, offline-tolerant, `src/shell/autonomy-surfacing.ts`): `LastSeenStore` refreshes every 5 min; at launch **"While you were away" digest** (`src/core/away-digest.ts` — schedules that ran, tasks completed, deliveries since lastSeen); sidebar **"Coming up"** merges daemon job `nextRunAt` with local calendar events (60 s TTL cache, 2.5 s fetch timeout, silent-failure); each away-digest pass also runs the **skill-draft proposer** (auto-accrues skill drafts from observed usage). Calendar boot refresh: `src/runtime/calendar-boot-refresh.ts`.
- **Learning loop** — curator proposes memory/behavior candidates from reviewed notes, completed work, completed research, saved sessions; duplicate consolidation with receipts; only reviewed/high-confidence records enter prompts (suppressed work is surfaced, never silently applied).

## 5. Configuration + Observability Surfaces

**Configuration.** SDK `ConfigManager` with schema (`CONFIG_SCHEMA`/`DEFAULT_CONFIG` from `platform/config`); settings routes: `/settings`, `/config` (fullscreen workspace), `/setup`, model tools `settings action:"list|get|set|reset|import"`; secret-backed values stored via `SecretsManager` (project + global home, `preferred_secure` policy) and always redacted (`src/config/secrets.ts`, `secret-config.ts`). Env/flags: `GOODVIBES_AGENT_HOME`, `--agent-profile`, `GOODVIBES_AGENT_RUNTIME_URL`/`--runtime-url`, `GOODVIBES_WORKING_DIR`; endpoint config keys `controlPlane.hostMode/host/port` (`src/cli/endpoints.ts`); email keys `email.imap*/smtp*`; `automation.enabled` (ignored locally with warning); keybindings file `agent/keybindings.json` writable via `/keybindings` and `workspace action:"set_keybinding"`. Settings policy layer: `src/config/agent-settings-policy.ts` (connected-host lifecycle/listener settings read-only). Opt-in local network scan for model servers: `/network-scan` (never scans silently). Shared settings import from TUI: preview→confirm, ownership metadata, redaction.

**Observability.** 
- CLI: `goodvibes-agent status [--json]`, `doctor`, `compat` (`src/cli/status.ts`, `service-posture.ts`).
- TUI: `/health`, `/compat`, `/auth`, `/brief`, `/context`, `/conversation`, `/session`, `/security`, `/trust`; Home → Host compatibility / Doctor diagnostics / Review health.
- Model-visible read-only diagnostics: `host action:"status|capabilities|services|methods"` (endpoint binding, probes, redacted log tail), `security action:"status|finding|explain"` (policy-explain for one tool action), `support action:"status|bundle"`, `audit action:"readiness|evidence|item|artifact"` (packaged release evidence — `release/release-readiness.json`, `release/live-verification/*` shipped in the npm package), `execution action:"status|history|record|processes"` (execution ledger — `src/runtime/execution-ledger.ts`), `context action:"receipts"` (prompt-build receipts), `channels action:"deliveries"` (redacted receipts), `memory action:"status"` (vector/embedding doctor).
- Support bundles: `/bundle` export/inspect/import, redacted (`src/cli/bundle-command.ts`).
- Internal: runtime diagnostics panels (`src/runtime/diagnostics/`), perf budgets (`src/runtime/perf/`, `scripts/perf-check.ts`), UI observability read-models re-exported from runtime (`src/runtime/ui-read-models-observability-{security,system,remote,maintenance}.ts`), component health monitor + focus tracker in the service graph.
- Release/quality gates in-repo: `ci:gate` chain (typecheck, tests, coverage gate, architecture check, perf check, publish check, install check, verification ledger — `package.json` scripts; 800-line file cap noted in `src/shell/autonomy-surfacing.ts`).

**Key implications for the new goodvibes-app:** the SDK already carries nearly all runtime logic (Orchestrator, services graph, typed daemon client with operator-method contracts, memory/session spines with local→REST activation); goodvibes-agent is essentially "SDK service graph + terminal renderer + Agent-brand registries/tools/confirmation-gate policy". A GUI app can reuse the same `createRuntimeServices`-style composition, the `~/.goodvibes/agent` (or new surface root) storage layout, the canonical shared memory DB, the 127.0.0.1:3421 daemon contract, and the confirm+explicitUserRequest effect-gating model wholesale.
